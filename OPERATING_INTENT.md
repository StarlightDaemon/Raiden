# Operating Intent

## Purpose

This file states why RAIDEN exists and how RAIDEN should interpret its mission
under resource-constrained working conditions.

Use it as the canonical mission-and-operating-intent layer for future RAIDEN
development.

## Problem Definition

RAIDEN exists because the current working environment fails in two directions:

- long chat sessions degrade under accumulated history
- agent tooling burns usage rapidly during active technical work

Observed failure patterns include:

- context drag
- redundancy
- repeated re-explanation
- repeated rediscovery of the same files and decisions
- broad context loading
- excessive setup overhead
- weak continuity between work cycles

RAIDEN must assume that token budgets are limited, model continuity matters, and
expensive abundance is not a safe default.

## Why RAIDEN Exists

RAIDEN is a durable operating foundation for reducing repeated waste across
technical work.

Its purpose is not only to complete the current task.
Its purpose is to reduce the amount of context, setup, and coordination that
must be recreated when work starts, resumes, pivots, or deepens.

RAIDEN should increase leverage by turning repeated work into reusable
infrastructure.

## Core Mission Implications

RAIDEN should prefer:

1. durable reusable assets over repeated ad hoc explanation
2. bounded work cycles over long mixed-purpose sessions
3. compact continuation state over raw history replay
4. exact scope over broad repo scanning
5. infrastructure-first fixes when repeated operator pain is observed

RAIDEN should assume:

- agents may be token-blind
- work cycles may be model-bound
- mid-cycle model switching may be unavailable
- repeated setup cost is harmful
- broad context reload is harmful
- raw chat accumulation is harmful

## Persistent Versus Transient

Persist durably:

- governance and policy
- stable repo guidance
- durable decisions
- reusable prompt and workflow assets
- stable read order and operating rules
- compact continuation standards
- validated task and handoff formats

Keep transient or local:

- session-shaped working prompts
- one-off execution scaffolds
- temporary planning notes
- local repo-specific overlays
- pause-point exports tied to one active work stream

## Human-Readable Versus Compressed

Keep human-readable:

- governance
- operator handoffs
- long-term notes
- adoption guidance
- final reports
- durable repo documentation

Allow compressed machine-oriented form:

- internal prompts
- scoped execution directives
- validation stubs
- pause-point packages
- continuation payloads
- compact review prompts
- transient working state

Compression is valid only when it does not reduce scope clarity, validation
quality, or continuation reliability.

## Waste Triggers

Repeated waste should trigger new infrastructure when work repeatedly shows:

- the same setup framing
- the same repo explanation
- the same command or convention restatement
- the same read-order reconstruction
- the same handoff gap
- the same continuity loss after pause or restart
- the same broad scans before narrow work
- the same re-litigation of settled context

These are signals to standardize, compress, persist, or redesign the workflow.

## Operating Priorities

Default priority order:

1. persist stable knowledge
2. narrow scope
3. compress context
4. separate phases
5. export compact continuation
6. reuse templates
7. avoid conversational sprawl
8. avoid full-history dependence
9. avoid brute-force token spending
10. preserve readability where durability requires it

## Repository Implications

For RAIDEN repo operations:

- keep root canon concise and durable
- move repeated execution patterns into reusable assets
- keep prompt and handoff standards centrally governed
- keep local or session-shaped state out of root canon
- prefer exact file targets and bounded read sets
- treat each repeated operator pain point as a candidate for infrastructure

## Immediate Use

Use this file when:

- defining new RAIDEN policy
- deciding whether a repeated pain point should become infrastructure
- deciding whether information should be persisted or left transient
- deciding whether a surface should stay readable or be compressed
- evaluating whether a workflow reduces future waste or only solves the current turn
